| 服务 | 端口 | 启动命令 |

### 配置确认

1. `config.yaml` 中 `oauth2.enabled` 设为 `true`
2. `oauth2.auth_url` 指向 `http://localhost:3000/oauth2/authorize`
3. `oauth2.redirect_url` 指向 `http://localhost:5173/auth/callback`
4. 环境变量 `DB_PASSWORD`、`OSS_ACCESS_KEY`、`OSS_SECRET_KEY` 已设置

## 测试依赖关系

测试按 T1→T15 顺序执行，存在数据依赖，不可乱序或单独运行中间用例。

```
T1 (登录 + 全局清理)
├── T2 (文件夹，自包含)
├── T3 (文件上传，创建 3 个测试文件)
│   ├── T4 (文件预览，依赖 T3 的文件)
│   ├── T5 (文件下载，依赖 T3 的文件)
│   ├── T7 (标签管理，依赖 T3 的文件)
│   ├── T9 (分享管理，依赖 T3 的文件)
│   └── T10 (权限管理 + 清理 T3 文件，依赖 T3 的文件，末尾删除全部文件和回收站)
├── T6 (回收站，自建测试文件)
├── T8 (版本回滚，自建测试文件)
├── T11 (存储空间，自建测试文件)
├── T12 (网关管理，独立)
├── T13 (跨用户隔离，独立)
└── T14 (页面导航，独立)
T15 (最终验证，检查数据污染)
T16 (网关 Agent 管理，独立)
T17 (Chat ReturnFile + Markdown 渲染，独立)
T18 (Admin Bootstrap，独立)
T19 (公共目录 + Admin 管理 + API Key，依赖 T18)
```

**清理规则**：
- T1 负责全局清理（删除所有文件/文件夹 + API 清理分享/权限/回收站），为后续用例提供干净初始状态
- T3 上传的文件由 T10 在权限测试完毕后统一删除并清理回收站
- 各用例验证完毕后自行清理创建的数据，避免污染后续用例
- T15 不做清理，仅验证所有数据已归零（回收站、分享、权限、文件、文件夹）
